¿Qué es el MVP en términos de sistema?

El MVP no es solo "la versión más chica". Es la intersección entre lo más valioso para el usuario y lo más fácil de construir primero. Todo lo que no pertenece a esa intersección va al backlog.

✅
Regla de oro del MVP

Si dudás si algo va en el MVP preguntate: ¿el sistema funciona sin esto? Si la respuesta es sí, no va en el MVP.

Visión de Producto

El "para qué" del sistema. Define el problema que resuelve y a quién le resuelve.

Core del Sistema

El motor central: la funcionalidad sin la cual el sistema no tiene sentido de existir.

MVP

La versión mínima que entrega valor real. No un prototipo, no un mockup: algo que funciona.

Arquitectura Monolítica

Todo en un solo deployment. Ideal para MVPs: simple de entender, desplegar y mantener.

¿Por qué empezamos con arquitectura monolítica?

Muchos sistemas que hoy son microservicios empezaron siendo monolitos: GitHub, Shopify, Airbnb. La arquitectura monolítica es la base correcta cuando el equipo es pequeño, el dominio no está bien definido y necesitamos velocidad de desarrollo.

CriterioMonolitoMicroservicios
Complejidad inicial🟢 Baja🔴 Alta
Velocidad de desarrollo🟢 Alta🟡 Media
Equipo pequeño🟢 Ideal🔴 Problemático
Escalabilidad futura🟡 Posible🟢 Alta
Adecuado para MVP🟢 Sí🔴 No recomendado

02 · Estructura

Estructura del proyecto

Toda aplicación web real necesita una estructura de carpetas consistente. No es capricho: es la diferencia entre un proyecto que escala y uno que se convierte en caos a la semana 3.

📁 mi-proyecto/
  📁 frontend/  ← Todo lo que ve el usuario
    📁 src/
      📁 components/  ← Componentes React reutilizables
      📁 pages/  ← Vistas / páginas completas
      📁 services/  ← Llamadas al backend (fetch/axios)
      📄 App.jsx  ← Componente raíz + rutas
    📄 package.json
    📄 vite.config.js

  📁 backend/  ← Lógica de negocio y API
    📁 routes/  ← Definición de endpoints
    📁 controllers/  ← Lógica de cada endpoint
    📁 models/  ← Modelos de datos / acceso a DB
    📁 middleware/  ← Auth, validaciones, logs
    📄 index.js  ← Punto de entrada del servidor
    📄 package.json

  📁 database/  ← Modelo de datos
    📄 schema.sql  ← Estructura de tablas
    📄 seed.sql  ← Datos iniciales para pruebas
    📄 diagrama_ER.png  ← Diagrama entidad-relación

  📁 docs/  ← Documentación del proyecto
    📄 vision.md  ← Visión, core, MVP
    📄 epicas.md  ← Épicas e historias de usuario
    📄 api.md  ← Documentación de endpoints

  📄 README.md  ← Descripción general + cómo levantar
  📄 .gitignore
⚠️
Nunca commiteen node_modules, .env ni archivos de contraseñas

El archivo .gitignore debe incluir: node_modules/, .env, *.log, dist/

¿Qué va en cada capa?

Frontend (React)

Interfaz de usuario. Consume la API del backend. No tiene lógica de negocio ni acceso directo a la DB.

Backend (Node/Express)

API REST. Contiene la lógica de negocio. Es el único que habla con la base de datos.

Database (MySQL/PostgreSQL)

Almacena los datos de forma persistente. El schema define la estructura del sistema.

Docs

Documentación viva del proyecto: visión, épicas, endpoints. Tan importante como el código.

Un sistema simple de Lista de Tareas (todos) con React + Node + MySQL. El ejemplo más simple que demuestra el flujo completo.

// backend/routes/tareas.js
const express    = require('express');
const router     = express.Router();
const controller = require('../controllers/tareasController');

// GET  /api/tareas        → listar todas
router.get('/',       controller.getAll);

// GET  /api/tareas/:id    → obtener una
router.get('/:id',    controller.getOne);

// POST /api/tareas        → crear
router.post('/',      controller.create);

// PUT  /api/tareas/:id    → actualizar
router.put('/:id',    controller.update);

// DELETE /api/tareas/:id  → eliminar
router.delete('/:id', controller.remove);

module.exports = router;

🎮 Backend — controllers/tareasController.js

javascript · controller 
// backend/controllers/tareasController.js
const db = require('../models/db');

exports.getAll = async (req, res) => {
  try {
    const [rows] = await db.query('SELECT * FROM tareas ORDER BY created_at DESC');
    res.json(rows);
  } catch (err) {
    res.status(500).json({ error: 'Error al obtener tareas' });
  }
};

exports.create = async (req, res) => {
  const { titulo, descripcion } = req.body;

  // Validación básica
  if (!titulo) {
    return res.status(400).json({ error: 'El título es obligatorio' });
  }

  try {
    const [result] = await db.query(
      'INSERT INTO tareas (titulo, descripcion) VALUES (?, ?)',
      [titulo, descripcion]
    );
    res.status(201).json({ id: result.insertId, titulo, descripcion });
  } catch (err) {
    res.status(500).json({ error: 'Error al crear tarea' });
  }
};

CREATE DATABASE IF NOT EXISTS mi_app_db;

CREATE TABLE usuarios (

  nombre      VARCHAR(100) NOT NULL,

  password    VARCHAR(255) NOT NULL,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE tareas (
  id           INT AUTO_INCREMENT PRIMARY KEY,
  usuario_id   INT,
  titulo       VARCHAR(200) NOT NULL,
  descripcion  TEXT,
  completada   BOOLEAN DEFAULT FALSE,
  created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (usuario_id) REFERENCES usuarios(id) ON DELETE CASCADE
);

⚛️ Frontend — App.jsx

jsx · react 
// frontend/src/App.jsx
import { useState, useEffect } from 'react';
import { getTareas, createTarea } from './services/tareaService';

export default function App() {
  const [tareas, setTareas]   = useState([]);
  const [titulo, setTitulo]   = useState('');
  const [loading, setLoading] = useState(true);

  // Cargar tareas al montar el componente
  useEffect(() => {
    fetchTareas();
  }, []);

  const fetchTareas = async () => {
    const data = await getTareas();
    setTareas(data);
    setLoading(false);
  };

  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!titulo.trim()) return;
    await createTarea({ titulo });
    setTitulo('');
    fetchTareas();
  };

  return (
    <div className="app">
      <h1>Mis Tareas</h1>

      <form onSubmit={handleSubmit}>
        <input
          value={titulo}
          onChange={(e) => setTitulo(e.target.value)}
          placeholder="Nueva tarea..."
        />
        <button type="submit">Agregar</button>
      </form>

      {loading ? <p>Cargando...</p> : (
        <ul>
          {tareas.map(t => (
            <li key={t.id}>{t.titulo}</li>
          ))}
        </ul>
      )}
    </div>
  );
}

🔌 Frontend — services/tareaService.js

javascript · fetch api 
// frontend/src/services/tareaService.js
const BASE_URL = 'http://localhost:3001/api';

export const getTareas = async () => {
  const res = await fetch(`${BASE_URL}/tareas`);
  return res.json();
};

export const createTarea = async (data) => {
  const res = await fetch(`${BASE_URL}/tareas`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(data)
  });
  return res.json();
};
💡
Flujo de una request completa

Usuario llena el form → React llama a createTarea() → fetch hace POST a /api/tareas → Express recibe la request → el controller valida y ejecuta el INSERT → MySQL guarda el dato → el controller responde JSON → React actualiza la lista.

05 · Inteligencia Artificial

IA aplicada a su sistema

La IA no es magia ni hay que tenerle miedo. Es una herramienta. Lo importante es dónde aplicarla, cómo controlarla y cuándo el humano debe tomar el control.

Repaso rápido del ecosistema

Machine Learning (ML)

El sistema aprende de datos históricos. Ejemplo: predecir qué categoría corresponde a una tarea.

Deep Learning (DL)

ML con redes neuronales profundas. Útil para texto, imágenes y voz.

MLP (Perceptrón Multicapa)

Red neuronal básica, feedforward. El building block de las redes profundas.

LLMs (GPT, Claude, etc.)

Modelos de lenguaje. Pueden usarse vía API para generar texto, clasificar, resumir.

🛡️ Guardrails — Reglas para usar IA responsablemente

Un guardrail es una regla o límite que le ponemos al sistema de IA para que no haga cosas que no queremos.

🚫

Guardrail 1 — No mostrar resultados sin mínimo de confianza: Si el modelo no está seguro (confianza < 70%), mostrar "No podemos sugerir nada todavía" en vez de una respuesta incorrecta.

👤

Guardrail 2 — Human-in-the-loop para decisiones críticas: La IA sugiere, el humano decide. Ejemplo: la IA detecta una anomalía médica → alerta al usuario → el usuario confirma si desea continuar.

🔍

Guardrail 3 — Transparencia: Siempre informar al usuario que está interactuando con IA. "Sugerido por IA" debe verse claramente en la UI.

📝

Guardrail 4 — Log de decisiones: Guardar en base de datos qué sugirió la IA, cuándo y qué eligió el usuario. Permite auditoría y mejora del modelo.

🔒

Guardrail 5 — Nunca enviar datos sensibles a APIs externas: Si usan una API de IA externa (como OpenAI), no envíen contraseñas, DNI, datos médicos completos, o información privada sin consentimiento.

Ejemplo simple — Sugerencia de prioridad con API de IA

javascript · llamada a api de ia 
// backend/services/iaService.js
// Ejemplo conceptual: usar una API de IA para sugerir prioridad

async function sugerirPrioridad(textoTarea) {
  // Solo enviamos el texto de la tarea, nunca datos del usuario
  const response = await fetch('https://api.ejemplo-ia.com/classify', {
    method: 'POST',
    headers: { 'Authorization': `Bearer ${process.env.IA_API_KEY}` },
    body: JSON.stringify({ text: textoTarea })
  });

  const data = await response.json();

  // Guardrail: solo usamos la sugerencia si la confianza es alta
  if (data.confidence < 0.70) {
    return { sugerencia: null, mensaje: 'No hay suficiente información para sugerir' };
  }

  return {
    sugerencia: data.label,        // 'alta', 'media', 'baja'
    confianza: data.confidence,
    esSugerencia: true            // Siempre marcamos que es IA
  };
}
